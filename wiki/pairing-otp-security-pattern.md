# Pairing/OTP 승인 시스템: 목록에 보이는 값이 실제 비밀 코드가 아닐 수 있다

pairing/OTP류 승인 시스템을 만들거나 운영할 때, "관리자 목록 화면에 보이는 식별자"와 "사용자에게
전달되는 실제 비밀 코드"를 다른 값으로 분리하는 건 흔한 보안 패턴이다 (관리자 화면 캡처/로그 유출만
으로는 승인을 못 하게 막기 위함).

## 증상과 원인

낯선 사용자가 봇에 말을 걸면 pairing 코드가 발급되고, 관리자가 대기 목록 명령으로 코드를 볼 수
있었다. 그런데 목록에 나온 값을 그대로 승인 명령에 넣으면 "not found or expired" 에러가 났다 —
버그가 아니라 의도된 설계였다:

- 실제 코드는 salt+SHA256으로 해시되어 저장된다.
- 목록이 보여주는 값은 **해시값의 앞 8자리(fingerprint)** 일 뿐, 관리자가 항목을 구분하도록
  보여주는 용도이지 원본 코드가 아니다.
- 진짜 코드는 요청자 본인의 화면에만 표시되며, 그 사람이 직접 운영자에게 알려줘야 승인이 가능하다.

**교훈**: 이런 시스템을 처음 쓸 때 "목록에 나온 값을 그대로 넣었는데 안 된다"는 증상을 보면 버그
의심보다 먼저 "이 값이 fingerprint인지 실제 secret인지"를 소스에서 확인하는 게 빠르다.

비슷한 이유로 신중하게 다뤄야 한다는 사례가 있다.
