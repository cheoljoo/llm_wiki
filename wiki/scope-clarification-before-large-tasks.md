# 작업량이 크게 갈리는 갈림길에서는 시작 전에 스코프를 확인한다

"Makefile에 있는 각각이 어떤 의미인지 주석으로 달아달라"는 요청을 문자 그대로 받으면 target 150개
전부에 개별 설명을 달아야 하는 것처럼 보였다(그중 109개는 이미 `##` 도움말 주석이 있었음). 하지만
실제로는 17개 섹션으로 나뉜 구조라 "각각"이 target 단위인지 섹션 단위인지에 따라 작업량이 10배 이상
차이 났다.

**판단**: 착수 전에 `AskUserQuestion`으로 "섹션 단위 요약 / target 개별 상세 / 특정 파일 관련 target만"
세 가지 선택지를 제시해 스코프를 "섹션 단위 요약"으로 확정한 뒤 시작했다. **작업량이 크게 갈리는
갈림길에서는, 시작 전에 한 번 물어보는 비용이 잘못된 방향으로 다 만들고 되돌리는 비용보다 훨씬
싸다.**

## 기존에 잘 된 부분을 스타일 가이드로 삼는다

파일 안에 이미 섹션 헤더 바로 아래 2~4줄짜리 설명이 잘 되어 있는 섹션들이 있었다. 이를 건드리지 않고,
설명이 없던 나머지 섹션에 **같은 톤과 분량**으로 맞춰 추가했다 — "기존에 잘 된 부분을 스타일 가이드로
삼아 새 부분을 맞춘다"는 원칙은 코드 리팩터링뿐 아니라 문서/주석 작업에도 그대로 적용된다.

## 가장 크고 복잡한 부분은 하위 그룹으로 쪼개서 설명한다

가장 큰 섹션은 실제로는 서로 다른 4가지 워크플로가 하나의 헤더 아래 뭉쳐 있었다. 한 문단으로
뭉뚱그리는 대신, 섹션 상단에 하위 그룹들의 관계(한 그룹의 출력이 다른 그룹의 입력이 되는 파이프라인
구조 등)를 먼저 요약하고, 각 하위 그룹 시작점에 짧은 구분 주석을 추가해 "긴 섹션 안에 숨은 하위 구조"를
드러냈다. 섹션 헤더와 하위 구분 주석을 시각적으로 다른 굵기로 표기해 계층을 구분하면 나중에 훑어볼 때
도움이 된다.

## 공백/탭에 민감한 파일은 주석만 추가해도 dry-run으로 검증한다

Makefile처럼 공백/탭에 민감한 파일에 `#` 설명만 추가하고 `##`(도움말 주석)은 건드리지 않았더라도,
`make help` 출력이 그대로인지, `make -n <target>` 몇 개가 문법이 깨지지 않았는지 반드시 dry-run으로
확인하는 습관이 안전하다.
