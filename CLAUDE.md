# llm_wiki 저장소 작업 지침

이 저장소는 "원본 로그 → 정제된 위키" 2단계 구조로 지식을 축적합니다.

## 구조

- `log/YYYY-MM-DD-<project>-<slug>.md`: append-only 원본 기록. 새 파일만 추가, 기존 파일은 `digested` 필드만 수정.
- `wiki/<topic>.md`: 주제별 정제 문서. 여러 log 항목의 내용이 여기로 통합됨.
- `wiki/README.md`: 전체 주제 목차 (digest 시 자동 갱신). 카테고리(`##` 섹션)별로 묶여 있으며,
  통째로 읽기보다 키워드로 grep해서 찾도록 되어 있다 — 주제가 늘어나도 확장성 있게 쓰기 위함.

## `/wiki-digest` 실행 시 규칙

1. `log/*.md` 중 frontmatter `digested: false`(또는 없음)인 파일만 대상으로 한다. 이미 `digested: true`인 파일은 건드리지 않는다.
2. 각 로그의 내용을 검토해 어떤 `wiki/<topic>.md`에 속하는지 판단한다. 기존 주제와 겹치면 새 파일을 만들지 말고 기존 파일을 갱신한다.
3. 위키 문서는 "정제된" 지식만 담는다 — 로그의 잡담이나 세션 진행 상황이 아니라, 재사용 가능한 결론·패턴·트러블슈팅 노하우·설계 결정과 그 이유만 남긴다.
4. 관련 있는 다른 주제 문서는 `[[topic-name]]`으로 상호 링크한다.
5. 처리한 로그 파일은 frontmatter의 `digested`를 `true`로 바꾼다 (파일을 지우거나 옮기지 않는다 — 원본 기록은 보존).
6. `wiki/README.md`의 목차를 새/변경된 주제에 맞게 갱신한다. 맞는 카테고리(`##` 섹션) 아래에 넣고,
   어느 카테고리에도 안 맞으면 새 섹션을 만든다 — 목차가 계속 자라도 grep으로 찾기 쉽게 유지하기 위함.
7. 변경사항을 하나의 커밋으로 남긴다 (`git add log wiki && git commit`). push는 하지 않는다 — 사용자가 검토 후 직접 push.

## 새 주제 문서(`wiki/<topic>.md`) 작성 규칙

- 맨 위에 한 줄 요약.
- "왜"가 비직관적인 경우에만 이유를 남긴다 (당연한 내용은 생략).
- 코드/파일 경로를 인용할 때는 어느 프로젝트(log의 `project` frontmatter)에서 나온 내용인지 각주로 남긴다 — 위키만 보고도 원 출처를 추적할 수 있어야 한다.

## 다른 프로젝트에서 이 위키 활용하기

`log/`는 쓰기 전용(append-only) 저장소이고, 실제로 다시 참고하는 대상은 `wiki/`다. 활용 방법:

- 수동: 새 작업 시작 전 `wiki/README.md`를 통째로 읽게 하지 말고, 관련 키워드로 grep해서 일치하는
  주제만 찾아 그 파일을 읽어달라고 요청한다 (예: `grep -i docker <WIKI_REPO_PATH>/wiki/README.md`).
  목차가 카테고리별로 정리돼 있어 키워드 grep이 잘 먹는다. 주제 수가 늘어나도 README.md를 전체
  읽는 것보다 비용이 거의 늘지 않는다.
- 자동화: 해당 프로젝트의 CLAUDE.md에 "X 작업 전에는 `<WIKI_REPO_PATH>/wiki/<topic>.md`를 참고하라"는
  지침을 추가해두면, 세션마다 알아서 읽고 시작한다 (이 경우는 대상 파일이 이미 특정돼 있으니 grep이
  필요 없다).

`wiki/`가 늘 최신 상태를 유지하려면 log가 어느 정도 쌓일 때마다 `/wiki-digest`를 주기적으로 실행해야 한다 — 너무 자주 돌리면 정제 품질(중복 제거, 주제 분류)이 떨어지므로 매번 돌릴 필요는 없다.
